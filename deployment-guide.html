Application Server Deployment Guide

WildFly + Maven on Red Hat / CentOS Linux

by Vyshnavi Gurram – Technical Writer

Introduction

This sanitized sample provides step-by-step procedures and scripts to deploy a Java application server on Red Hat and CentOS Linux using WildFly and Maven. It uses placeholders for application packages, database values, server paths, and credentials so the guide can be shared publicly.

Minimum Software Requirements

  • OS: CentOS 7+ Minimal Installation
  • DB: Oracle-compatible database

Sample Deployment Requirements

Step 1 — Installing JDK 8

bash 
# cd <DEPLOYMENT_DIR>
# ./jdk-install.sh
Note

Do not un-tar the zip file. If a previous version of Java is installed, you will be prompted to select a version. Select /opt/jdk1.8.0_131/bin/java.

Once setup is complete, you will see: "Java 8 (64-bit) setup is complete".

Verify the installed version:

bash
# java -version
java version "1.8.0_131"
Java(TM) SE Runtime Environment (build 1.8.0_131-b11)
Java HotSpot(TM) 64-Bit Server VM (build 25.131-b11, mixed mode)

Step 2 — Installing WildFly

Part A — Install WildFly 10.1.0.Final

bash 
# ./wildfly-install.sh

Type yes to continue and wait for the installation to complete successfully.

Part B — Add User to WildFly Management

bash 
# cd /opt/wildfly-10.1.0.Final/bin
# ./add-user.sh

Leave the group field blank and press Enter to add the user to the management group by default.

Step 3 — Installing Maven 3.5.0

bash 
# cd <DEPLOYMENT_DIR>
# ./maven-install.sh

Wait until you see the success message. Maven will be enabled on restart.

bash 
# sudo reboot

After rebooting, verify Maven:

bash
# mvn -version

Step 4 — Installing the OJDBC Driver

This can be done either through Maven or the WildFly management console. The following instructions use Maven.

Upload ojdbc8.jar to the server, navigate to its directory, and run:

bash 
mvn wildfly:deploy-only \
  -Dwildfly.hostname=<SERVER_HOST> \
  -Dwildfly.username=<ADMIN_USER> \
  -Dwildfly.password=<ADMIN_PASSWORD> \
  -Dwildfly.deployment.filename=ojdbc8.jar \
  -Dwildfly.deployment.targetDir=.

Wait for the BUILD SUCCESS message, then restart the service:

bash 
# service wildfly stop
# service wildfly start

Step 5 — Install LibreOffice

LibreOffice is used by the application to add and modify documents. Install using one of the following methods:

Option A — Install via yum
bash 
yum install openoffice.org-headless openoffice.org-writer \
            openoffice.org-calc openoffice.org-impress openoffice.org-draw
Option B — Install from downloaded RPM
bash 
wget <DOWNLOAD_URL>/LibreOffice_6.0.4_Linux_x86-64_rpm.tar.gz
tar -xvf LibreOffice_6.0.4_Linux_x86-64_rpm.tar.gz
cd LibreOffice_6.0.4.2_Linux_x86-64_rpm/RPMS
yum install *.rpm

Step 6 — Install Wkhtmltopdf

bash 
wget https://github.com/wkhtmltopdf/wkhtmltopdf/releases/download/0.12.4/wkhtmltox-0.12.4_linux-generic-amd64.tar.xz
sudo tar xvf wkhtmltox-0.12.4_linux-generic-amd64.tar.xz
sudo mv wkhtmltox/bin/wkhtmlto* /usr/bin/
yum install libXrender libXext fontconfig    # install dependencies

Step 7 — Increase WildFly Heap and Thread Usage

To handle the application load properly, increase WildFly's memory allocation:

  1. Log in to the server.
  2. Stop the WildFly service.
  3. Navigate to /opt/wildfly-10.1.0.Final/bin/ and open standalone.conf for editing.
  4. Set -Xmx1G and -XX:MaxMetaspaceSize=1G.
  5. Start the WildFly service.

Step 8 — Database Configuration

Additional tables, sequences, triggers, and views need to be added to the existing database using the provided schema.sql script.

Check Container Name

sql 
sqlplus / as sysdba
column name format a30
select con_id, name, open_mode from v$pdbs;

Check Schema Name

sql 
alter session set container=<CONTAINER_NAME>;
select username from dba_users order by 1;

Run the SQL Script

sql 
sqlplus <DB_USER>/<DB_PASSWORD>@<DB_SERVICE>
@<DB_SCHEMA_SCRIPT>.sql
exit
Important

If the application requires seed or reference data, migrate it using a sanitized data script. Do not publish internal table names, production records, or business-specific reference data in a public portfolio sample.

Step 9 — Add Datasource to Application Server

Open the WildFly Management Console and navigate to Configuration > Subsystems > Datasource > Non-XA, then click Add and select Oracle Datasource.

Datasource Attributes
config
Name:             APP_DATASOURCE
JNDI:             java:jboss/datasources/APP_DATASOURCE
Is Enabled:       true
Driver:           ojdbc8.jar
Driver Class:     oracle.jdbc.OracleDriver
Connection Settings
config
Connection URL:   jdbc:oracle:thin:@//<DB_HOST>:<DB_PORT>/<DB_SERVICE>
Use JTA:          true
Use CCM:          true
Security Settings
config
Username:   <DB_USER>
Password:   <DB_PASSWORD>

Once the datasource is added, restart the service and test the database connection:

bash 
# service wildfly stop
# service wildfly start

Step 10 — Add Default System Properties

In the WildFly Management Console, navigate to Configuration > System Properties > View, then click Add for each of the following:

Key Value
app.jwt.secret <SECURE_SECRET_VALUE>
app.report.output.dir <REPORT_OUTPUT_DIR>
document.converter.path <LIBREOFFICE_INSTALL_PATH>
app.export.output.dir <EXPORT_OUTPUT_DIR>
security.service.host <SECURITY_SERVICE_HOST>
security.service.port <SECURITY_SERVICE_PORT>

Step 11 — Deploy Application Files

Deploy WEB_APP_NAME.war

bash 
mvn wildfly:deploy-only \
  -Dwildfly.hostname=<SERVER_HOST> \
  -Dwildfly.username=<ADMIN_USER> \
  -Dwildfly.password=<ADMIN_PASSWORD> \
  -Dwildfly.deployment.filename=WEB_APP_NAME.war \
  -Dwildfly.deployment.targetDir=.

Wait for the BUILD SUCCESS message before continuing.

Deploy APP_NAME.ear

bash 
mvn wildfly:deploy-only \
  -Dwildfly.hostname=<SERVER_HOST> \
  -Dwildfly.username=<ADMIN_USER> \
  -Dwildfly.password=<ADMIN_PASSWORD> \
  -Dwildfly.deployment.filename=APP_NAME.ear \
  -Dwildfly.deployment.targetDir=.

Wait for the BUILD SUCCESS message, then restart WildFly:

bash 
# service wildfly stop
# service wildfly start
Alternative

You can also deploy files using the WildFly Management Console by navigating to Home > Deployments > Add and browsing for the files.

Troubleshooting

Deployments most commonly fail during the application package deployment stage. Common causes include:

  • A missing required database object
  • Required seed or reference data that has not been migrated
  • General database configuration issues
  • Build errors
When a deployment fails

Start by reviewing the WildFly server logs and verifying the database schema and required reference data are complete and up to date.
