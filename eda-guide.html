Conceptual Guide

Event-Driven Architecture

Conceptual Guide & Glossary

by Vyshnavi Gurram – Technical Writer

Introduction

Event-Driven Architecture is a software design pattern where decoupled applications, microservices, and devices publish and subscribe to events asynchronously via an event broker. Publishers and subscribers are not coupled or dependent on each other — publishers do not need to know who or how many subscribers exist, and consumers do not need to know who the publishers are.

Glossary

Core terms to understand before diving into Event-Driven Architecture:

Event
A message sent to a topic that represents something that already happened. The publisher owns this message. These messages travel over a network to zero or multiple subscribers, but the publisher does not know about the subscribers.
Event Producer (Publisher)
The source system or service that identifies a change in state and publishes it. Example: a customer app sends an OrderPlaced event to a driver app and an admin app after an order is placed.
Event Consumer (Subscriber)
The system or service that reads messages from topics and reacts to them. Example: the driver app reads the OrderPlaced event and checks restaurant preparation times for pickup.
Event Broker (Route / Bus / Channel)
A bridge between producers and consumers. It receives events from the producer and delivers them to interested consumers. Example: Netflix acts as a broker that distributes films (events) to subscribers.
Topic
A logical channel where publishers send events of a specific type. Consumers subscribe to topics to receive events. Example: film genres like thrillers, comedies, and romances are topics.
Event Mesh
An interconnected network of event brokers that enables dynamic routing of events across hybrid-cloud or on-premises environments.
Eventual Consistency
A model where all consumers in a distributed system will eventually reflect the same state change, but not instantaneously. Example: Node 1 updates first; Node 2 follows — they agree eventually, but not simultaneously.
Idempotency
The ability to process the same event multiple times without changing the result beyond the first occurrence. Important for handling retries. Example: clicking Pay multiple times — the amount is only charged once.
Dead Letter Queue (DLQ)
A queue where the broker sends messages that consumers failed to process, even after multiple retry attempts.
Loose Coupling
Systems or services in the architecture have little to no dependency on each other.
Backpressure
The consumer's ability to pull or poll events from the broker at its own pace, preventing being overwhelmed by too many events at once.
Fan-out
Broadcasting a message published to a topic to multiple consumers simultaneously.
Orchestrator
A controller in the Mediator Topology that handles complex logic, sequences steps, and manages errors centrally.
Connector / Action
Built-in interfaces that allow brokers to communicate with external applications like Salesforce, Jira, or SharePoint.
REST API
A synchronous communication style that waits for a response before continuing — contrasted with EDA's asynchronous model.
Retry
The attempt by a broker or publisher to resend a message to a consumer that failed to process it the first time.

Conceptual Guide & Architectural Principles

The main concept of EDA is achieving the highest possible loose coupling — systems or services working independently. This independence allows them to scale and react to changes without coordinating with other services.

Asynchronous Communication

EDA enables asynchronous communication: publishers can send events and immediately continue working without waiting for a response, unlike REST APIs. The broker holds the event until consumers are ready to process it.

Real-world analogy

A movie producer sells a film to Netflix and does not wait for subscribers to respond before starting the next project. Netflix holds the film and streams it to subscribers when they are ready to watch.

EDA Topologies

Broker Topology

This pattern broadcasts events to all consumers regardless of whether they subscribed. Publishers have no knowledge of the consumers — consumers decide whether to act. This topology works well for simple, decentralised flows. Consumers pull or poll the broker at their own capacity, which protects them from being overwhelmed (backpressure).

Example

Netflix makes a new film available to all subscriber tiers. Acting on it (watching) is left to the subscriber.

Patterns under Broker Topology

Event Sourcing

Stores a sequence of events rather than just the final state, allowing the state to be replayed at any point in time.

Example: Instead of storing only OrderDelivered, the system stores the full sequence: OrderPlaced → PaymentConfirmed → FoodPrepared → DriverPickedUp → OrderDelivered.

Choreography

Services act autonomously without a central controller. Each service knows what to do when it receives an event, producing a coordinated "dance" of services.

Example: When an Amazon customer orders, the payment service handles payment, the shipping service ships the package, and the notification service sends emails — all without a central controller.

Event-Carried State Transfer

Events carry all the necessary information, allowing consumers to update their local state without querying the producer.

Mediator Topology

A central mediator (orchestrator) controls workflows, coordinates multiple steps, and handles errors centrally. This topology is beneficial for complex, centralised flows.

Example

When a customer orders from an online bookstore, the mediator calls the payment service, inventory service, and shipping service in sequence.

CQRS (Command Query Responsibility Segregation)

Separates the write (command) model from the read (query) model, allowing them to scale independently. Most commonly used with Mediator Topology, where complex write flows benefit from centralised coordination.

Pattern comparison

Pattern Topology Notes
Event Sourcing Both Usually works independent of topology.
Choreography Broker No central controller.
Event-Carried State Transfer Both Helps reduce query load.
CQRS Both (most common with Mediator) Complex write flows benefit when paired with Mediator.

Event Processing Models

Simple Event Processing

A single event triggers an immediate action. Common in both topologies. Example: OrderPlaced triggers SendConfirmationEmail.

Event Stream Processing (ESP)

A stream of data is analysed in real-time to detect trends. Example: a real-time stock market analysis dashboard.

Complex Event Processing (CEP)

The consumer processes multiple, independent events to identify patterns or complex situations. Example: detecting fraud based on location and transaction volume.

Note

ESP and CEP pair well with Broker Topology because they need to analyse large streams of events that a broker distributes.

Benefits & Challenges

Benefits

  • Loose Coupling: Publishers, subscribers, and services can all be developed, deployed, and scaled independently.
  • Scalability: Adding more consumers for parallel processing (fan-out) is straightforward without crashing other services.
  • Resiliency: Failures in one consumer do not stop the producer or other consumers.
  • Independent Evolution: Teams can move at different speeds without coordinating every code change.

Challenges

  • Operational Complexity: Debugging and tracing event flows across many services and nodes is difficult.
  • Eventual Consistency: Nodes may temporarily disagree on the current state — the UI must handle stale data gracefully.
  • Event Ordering: Never assume events sent in one order will arrive in the same order. Ensuring correct sequence requires careful design (partitioning).

Conclusion

Event-Driven Architecture is the right choice when large volumes of data need to be handled reliably without relying on synchronous request-response patterns. It excels when systems need to react to changes in real time while remaining independently deployable — as seen in e-commerce platforms like Amazon, food delivery apps like UberEats, and real-time financial dashboards.

EDA is not the right fit for simple applications with few services and small data volumes. The trade-offs of eventual consistency and operational complexity are real — synchronous architectures avoid both of those challenges and remain the better choice at smaller scale.
