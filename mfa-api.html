Multi-Factor Authentication API

by Vyshnavi Gurram – Technical Writer

Note

Mock API Reference, built to demonstrate API documentation skills. All endpoint behaviours, values, and error codes are consistent with the accompanying Multi-Factor Authentication & Security Guide.

API Overview

The Multi-Factor Authentication API helps developers integrate user self-registration and login for banking applications, supporting OTP verification and TOTP-based MFA.

This API supports

  • Customer onboarding and account activation
  • Secure user login and authentication flows
  • Multi-factor authentication enrollment and management
  • Credential recovery and account access restoration

Use Cases

  • Banking platforms
  • Finance platforms
  • Payments platforms

Built on RESTful principles with JSON request/response bodies.

Key Features

  • Customer self-registration for existing account holders
  • Email-based OTP verification with rate limiting and account lockout protection
  • Multi-Factor Authentication (MFA) setup using QR codes (TOTP)
  • Secure login with TOTP verification
  • Password reset and recovery flows
  • Rate limiting and pagination support

Base URL

https://api.bankname.com/v1

Example Endpoint

https://api.bankname.com/v1/auth/mfa/verify

Authentication

The MFA API uses Bearer Token Authentication. Developers authenticate requests using an API key. The authToken should be included in the Authorization header for all subsequent authenticated user requests.

Example Header

http 
Authorization: Bearer YOUR_API_KEY
Example Request – curl
bash 
curl https://api.bankname.com/v1/auth/mfa/verify \
  -H "Authorization: Bearer YOUR_API_KEY"

Obtain API Key

Developer access to the MFA API is granted through the Bank Developer Portal.

  1. Log in to the Bank Developer Portal
  2. Navigate to API Keys
  3. Generate a new key for the application
  4. Copy and store the key securely — it will not be shown again

Example Key Format

text
bn_live_abc123xyz456def789
Caution

Never expose the API key in client-side code or public repositories. Store it as an environment variable.

Getting Started

The self-registration and MFA setup flow follows a sequential path. Each step depends on a token returned by the previous step and cannot be completed out of order.

1
Validate Customer Identity

Confirm the user is an existing customer.

2
Send Email Verification Code

Send a 6-digit OTP to the registered email.

3
Validate Email OTP

Verify the code the user entered.

4
Create Username and Password

Set credentials for the new account.

5
Generate MFA QR Code

Returned on first login — scan with authenticator app.

6
Verify TOTP Code

Complete MFA login; receive a session token.

Validate Customer Identity

Before registration begins, confirm the user is an existing customer by submitting their legal details.

POST/auth/validate-identity
Example Request – using account number
bash 
curl -X POST https://api.bankname.com/v1/auth/validate-identity \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "firstName": "Hannah",
    "lastName": "Barbara",
    "dateOfBirth": "2007-09-27",
    "accountNumber": "00123456789"
  }'
Example Request – using customer number
bash 
curl -X POST https://api.bankname.com/v1/auth/validate-identity \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "firstName": "Hannah",
    "lastName": "Barbara",
    "dateOfBirth": "2007-09-27",
    "customerNumber": "987654"
  }'
Example Response
json
{
  "status": "success",
  "registrationToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}
Caution

The registrationToken is valid for 10 minutes. Pass it to the next registration step. If it expires, the user must restart from Step 1.

Send Email Verification Code

Send a 6-digit OTP to the customer's registered email address. The code expires after 5 minutes.

POST/auth/send-verification
Example Request
bash 
curl -X POST https://api.bankname.com/v1/auth/send-verification \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "registrationToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "emailAddress": "h.barbara@email.com"
  }'
Example Response
json
{
  "status": "success",
  "message": "A 6-digit verification code has been sent to your registered email address.",
  "expiresInSeconds": 300
}

Response Codes

Code Meaning Description
200 OK Successful request The request was processed successfully.
201 Created Resource created A new resource was created successfully.
400 Bad Request Invalid request Missing required fields or contains invalid values.
401 Unauthorized Unauthorized The token is missing, invalid, or expired.
403 Forbidden Forbidden The account is suspended. Contact Customer Service.
404 Not Found Resource not found Identity details do not match any existing customer record.
409 Conflict Already exists The customer is already registered for digital banking.
423 Locked Session locked Too many failed OTP attempts. Session locked for 14 minutes.
429 Too Many Requests Rate limit exceeded Maximum send attempts exceeded. Retry after 840 seconds.
500 Internal Server Error Server error An unexpected server error occurred. Retry or contact support.

}
json · Rate Limit
{
  "status": "error",
  "code": "RATE_LIMIT_EXCEEDED",
  "message": "Maximum attempts exceeded.",
  "retryAfterSeconds": 840
}

Troubleshooting

This section helps developers resolve common issues during integration and testing.

401 Unauthorized

The request is missing a valid token or the token has expired.

  • Confirm the Authorization header is present and formatted correctly
  • Check that the token has not expired — authToken is valid for 30 minutes
  • If the registrationToken has expired, the user must restart from Step 1

Invalid TOTP Code

The most common cause is time drift on the user's device.

  • Ensure the device time is set to Automatic
  • The TOTP algorithm uses current UTC time — a clock offset of more than 30 seconds causes a code mismatch
  • If the issue persists, the user should re-scan the QR code after MFA is reset by Customer Service

OTP Not Received

  • Check the spam or junk folder
  • Wait 60 seconds before using the Resend Code option
  • Confirm the email address registered to the account is correct

423 Locked – Session Locked

  • The lock lasts 14 minutes (840 seconds)
  • The retryAfterSeconds field in the error response indicates remaining wait time
  • After the lock expires, the user may retry from Step 2

403 Forbidden – Account Suspended

The account is suspended after repeated failed MFA attempts. The user cannot retry and must contact Customer Service to have the MFA secret reset. They will need to re-scan a new QR code on next login.

Password Reset OTP Not Received

  • Check spam/junk folder
  • Ensure username/email is correct
  • Wait before requesting another OTP — rate limits apply

Migration Guide

Migrating from v1.0 to v1.1

Version 1.1 introduced optional trusted device support. No breaking changes were introduced.

Change introduced: deviceToken field added to /auth/mfa/verify — optional. Pass a previously issued device token to skip MFA on trusted devices.

Request comparison – v1.0 vs v1.1
json · v1.0 (still valid) 
{ "username": "h.barbara", "totpCode": "391847" }
json · v1.1 (with trusted device) 
{
  "username": "h.barbara",
  "totpCode": "391847",
  "deviceToken": "dev_Lp9mQrXt2s..."
}

Migrating from v1.1 to v1.2

Version 1.2 added webhook support. No changes to existing request or response formats.

  • Register a webhook endpoint in the Developer Portal to receive auth event notifications
  • Supported events: auth.registration.completed, auth.mfa.verified, auth.mfa.failed

Next Steps

Once the core authentication flow is integrated, consider extending the implementation with:

  • Webhook event listeners for real-time auth event handling
  • Trusted device token management for repeat users to reduce MFA friction
  • Session refresh flows to extend authenticated sessions beyond the 30-minute window
  • Audit logging for all authentication events to support compliance requirements
